===================================================
עיצוב מחלקות למערכת "הזמנות + תשלומים" (Java)
===================================================

עבור מערכת של הזמנות ותשלומים, אנו מזהים את הישויות המרכזיות ויוצרים חלוקת אחריות ברורה (Single Responsibility Principle).

1. אילו מחלקות הייתם יוצרים?

א. מחלקות מודל (Model Classes) - לייצוג נתונים:
----------------------------------------------
* Order: הזמנה ספציפית.
* OrderItem: פריט בודד בתוך ההזמנה (כמות + מוצר).
* Product: מוצר קיים (מחיר, שם, מלאי).
* Payment: טרנזקציית תשלום שבוצעה.
* Customer: פרטי הלקוח המזמין.

ב. מחלקות שירות (Service Classes) - לטיפול בלוגיקה עסקית:
------------------------------------------------------
* OrderService: טיפול ביצירה, עדכון ושליפה של הזמנות.
* PaymentService: טיפול בעיבוד תשלומים, אימות וקישור להזמנות.
* InventoryService: טיפול בבדיקת ועדכון מלאי.

2. מה האחריות של כל מחלקה?

| מחלקה | אחריות מרכזית |
| :--- | :--- |
| Customer | שמירת נתוני הלקוח. |
| Product | שמירת נתוני המוצר (כולל מחיר בסיס). |
| OrderItem | שמירת פרטים על פריט ספציפי בהזמנה (כמות, מחיר בעת ההזמנה). |
| Order | שמירת נתוני ההזמנה (מזהה, תאריך, סטטוס, רשימת פריטי הזמנה, סכום סופי). |
| Payment | שמירת נתוני התשלום (מזהה תשלום, סכום ששולם, שיטת תשלום, קישור להזמנה). |
| OrderService | חישוב סכום ההזמנה, יצירת הזמנה חדשה, עדכון סטטוס. |
| PaymentService | אימות תשלום, תקשורת עם ספק חיצוני, קישור התשלום להזמנה. |
| InventoryService | הורדת מלאי לאחר הזמנה, בדיקת זמינות. |

3. איך מוודאים שתשלום קשור להזמנה נכונה?

השיטה היא באמצעות שימוש ב'מפתח זר' (Foreign Key) בתוך מחלקת Payment, המצביע על מזהה ההזמנה:

* **מחלקת Order:** מכילה מזהה ייחודי (Primary Key) - `orderId`.
* **מחלקת Payment:** מכילה שדה `orderId` - שהוא המפתח הזר, המקשר אותה באופן ישיר להזמנה.
